# ТЗ ДЛЯ АГЕНТА — ВЕРСИЯ 2 (ДОПОЛНЕННАЯ)

## Разработка современного сайта компании и LMS-платформы для онлайн-обучения по ТБ

| Поле | Значение |
|---|---|
| Заказчик | ТОО / обучающая организация «Озен-пром» |
| Формат документа | Инженерное ТЗ для проектирования, реализации MVP и дальнейшего масштабирования |
| Версия | 2.0 — дополненная и уточнённая |
| Ключевая цель | Автоматизировать прием Excel-списков, создание пользователей, назначение курсов по должностям, прохождение тестов и экспорт итогов |

---

## 1. Резюме проекта

Нужно разработать систему из трёх связанных частей: публичный сайт компании, LMS для обучающихся и административная панель для массовой обработки потоков обучения.

Система ориентирована на реальный B2B-процесс: заказчик (организация-клиент) присылает Excel со списком своих сотрудников; администратор Озен-пром загружает файл; система автоматически создаёт учётные записи, генерирует логины и пароли, назначает отдельные курсы и сохраняет результаты прохождения.

Продукт не должен выглядеть как шаблонный лендинг или набор хаотичных CRUD-страниц. Нужен современный, быстрый и управляемый веб-сервис с чёткой архитектурой.

## 2. Роли пользователей системы

В системе три типа пользователей:

| Роль | Описание |
|---|---|
| `admin` | Сотрудник Озен-пром. Полный доступ: импорт, курсы, правила, пользователи, экспорт. |
| `learner` | Обучающийся (сотрудник организации-клиента). Доступ только к своим назначенным курсам. |
| `client_manager` *(опционально на MVP, заложить в схему БД)* | Представитель организации-клиента. Видит только своих сотрудников и их прогресс. Может скачать отчёт по своей организации. На MVP можно не реализовывать интерфейс, но таблица `organizations` и связь `user.organization_id` должны присутствовать с первого дня. |

> **Важно:** роль `client_manager` на MVP не реализуется как интерфейс, но модель данных должна её поддерживать. Это предотвратит переписывание ядра при расширении.

## 3. Бизнес-контекст и целевой процесс

### 3.1. Входные данные от заказчика

- Заказчик присылает Excel со списком сотрудников.
- В Excel, как правило, есть поля: ФИО, должность/профессия, организация, а также отметки по дисциплинам обучения.
- У одного сотрудника может быть назначено несколько дисциплин.
- Формат колонок может отличаться от файла к файлу — парсер должен быть устойчив к этому.

### 3.2. Что должно происходить в системе

1. Администратор создаёт новый поток обучения (batch) и привязывает его к организации-клиенту.
2. Загружает Excel-файл от заказчика.
3. Система распознаёт колонки и формирует preview.
4. Показываются строки: корректно распознанные, строки с предупреждениями и строки с ошибками.
5. После подтверждения система нормализует ФИО, должность, организацию и дисциплины.
6. Создаются пользователи (с проверкой на дубликаты — см. раздел 11.1).
7. Генерируются логины и пароли.
8. Назначаются отдельные курсы по каждой дисциплине через справочник правил.
9. Если правило по должности не найдено — строка уходит в manual review.
10. Сохраняется лог импорта.
11. Администратор скачивает Excel с логинами и паролями.
12. Обучающийся входит с телефона или компьютера, смотрит материалы и проходит тест.
13. Администратор видит прогресс, результаты и выгружает Excel-отчёты.

## 4. Обязательные принципы архитектуры

- Никакой бизнес-логики в шаблонах, представлениях или фронтенде. Вся ключевая логика живёт в backend-сервисах.
- Импорт Excel должен быть устойчивым к неполным данным и лёгким отличиям формата.
- Должность и назначение курса нельзя хардкодить в одном месте. Нужен отдельный справочник правил.
- Один сотрудник = много отдельных назначений. Никаких "пакетных псевдокурсов" вместо нормальной модели.
- Пароли хранить только как hash (bcrypt).
- Файлы материалов и видео хранить отдельно от бизнес-данных.
- Все интерфейсы должны быть адаптивными и пригодными для телефона.
- Данные разных организаций должны быть изолированы на уровне БД (поле `organization_id`) и API (фильтрация в репозиториях, никогда не на уровне роутов).

## 9. Логика курсов и должностей

Все обучения реализованы по должностям. Это ключевое требование, а не опция.

- Нельзя делать один общий курс «БиОТ для всех».
- Нужно поддерживать отдельные программы по категориям должностей: водитель, слесарь, автоматчик, ИТР и т.д.
- То же правило должно допускать расширение на ПТМ, промбез и другие дисциплины.
- Выбор курса при импорте происходит через справочник правил назначения.

### 9.1. Справочник правил

Каждое правило — это запись в таблице `position_course_rules`:

```
discipline_id | normalized_position_keyword | course_id | priority
```

**Пример логики:**

```
discipline = "БиОТ"
position = "слесарь-ремонтник"

rule lookup =>
  1. Нормализовать position -> "слесарь ремонтник"
  2. Найти строки в position_course_rules где discipline = "БиОТ"
  3. Перебрать keywords по приоритету: если keyword входит в normalized_position -> взять course_id
  4. Если совпадение найдено -> создать assignment
  5. Если нет -> пометить строку как manual_review_required
```

Таблица правил редактируется через UI администратора. Не хардкодится в коде.

## 10. Генерация логинов и паролей

### 10.1. Логин

- Формат: `transliterated_surname + initials + 2-3 random digits`
- Пример: Алиев Абылай Илиясулы → `AliyevAI27`
- Проверка уникальности: если логин занят — увеличить цифры / добавить суффикс.

**Казахская транслитерация — кастомная таблица:**

Стандартные Python-библиотеки (`transliterate`, `unidecode`) не покрывают специфические казахские буквы. Необходима таблица:

```python
KAZ_TRANSLIT = {
    'ә': 'a',  'Ә': 'A',
    'ғ': 'gh', 'Ғ': 'Gh',
    'қ': 'q',  'Қ': 'Q',
    'ң': 'ng', 'Ң': 'Ng',
    'ө': 'o',  'Ө': 'O',
    'ұ': 'u',  'Ұ': 'U',
    'ү': 'u',  'Ү': 'U',
    'һ': 'h',  'Һ': 'H',
    'і': 'i',  'І': 'I',
    # стандартные русские буквы — через стандартную таблицу
}
```

Сначала применяется казахская таблица, затем стандартная русская транслитерация.

### 10.2. Пароль

- Случайные 6-7 цифр: `482915` или `7310642`
- В таблицу выгрузки пароль попадает в открытом виде **только в момент первоначального создания**
- В базе хранится только hash (bcrypt)
- После первого экспорта открытый пароль нигде не доступен — только сброс

## 11. Импорт Excel: пошаговый сценарий

1. Администратор выбирает или создаёт поток обучения (batch) с привязкой к организации.
2. Загружает Excel-файл.
3. Бэкенд парсит файл: определяет колонки (по заголовкам, нечёткое совпадение), нормализует строки.
4. Возвращается preview с тремя категориями строк:
   - ✅ Корректно распознаны, курс найден
   - ⚠️ Распознан, но курс не найден по правилам → manual review
   - ❌ Критическая ошибка строки (нет ФИО) → пропустить
5. Администратор может вручную откорректировать строки из manual review до подтверждения.
6. После подтверждения:
   - Нормализуются ФИО, должность, организация, дисциплины
   - Создаются пользователи (с проверкой на дубликаты — см. 11.1)
   - Генерируются логины и пароли
   - Назначаются курсы по правилам
   - Спорные строки уходят в `import_rows` со статусом `manual_review`
   - Сохраняется лог импорта
7. Администратор скачивает Excel с логинами и паролями (Экспорт №1).

### 11.1. Обработка дубликатов при повторном импорте

Дубликат определяется по совпадению: `normalized_full_name + organization_id`.

| Ситуация | Поведение |
|---|---|
| Пользователь уже существует, должность та же | Пропустить, не трогать, отметить в логе как `already_exists` |
| Пользователь существует, должность изменилась | Предупреждение в preview, требует ручного подтверждения. По умолчанию — пропустить, не перезаписывать |
| Пользователь существует, дисциплина уже назначена и пройдена | Пропустить назначение, отметить в логе |
| Пользователь существует, дисциплина не назначена | Добавить новое назначение |
| Пользователь не существует | Создать нового |

## 12. Предлагаемая схема БД

### Таблицы

| Таблица | Назначение |
|---|---|
| `users` | Обучающиеся (сотрудники клиентов) |
| `admin_users` | Администраторы Озен-пром |
| `organizations` | Организации-клиенты (обязательна с первого дня) |
| `training_batches` | Потоки / партии импорта / заявки, привязаны к `organization_id` |
| `disciplines` | Справочник дисциплин (БиОТ, ПТМ, промбез и т.д.) |
| `positions` | Справочник должностей (нормализованные категории) |
| `courses` | Курсы, привязаны к `discipline_id` |
| `course_materials` | Материалы курса (файл или ссылка, тип, путь/URL) |
| `tests` | Тест для курса: pass_score, max_attempts, time_limit_minutes |
| `test_questions` | Вопросы теста с вариантами ответов (JSONB или отдельная таблица) |
| `test_question_options` | Варианты ответов к вопросам |
| `user_course_assignments` | Назначения курсов пользователям, статус, batch_id |
| `test_attempts` | Попытки прохождения теста: статус, score, passed, started_at, finished_at |
| `test_attempt_answers` | Ответы в конкретной попытке |
| `position_course_rules` | Правила «discipline + keyword → course_id», с приоритетом |
| `import_rows` | Логи всех строк из Excel: статус, ошибки, user_id если создан |
| `certificates` | Заготовка под будущие удостоверения (id, user_id, assignment_id, issued_at, number) — не используется на MVP, но таблица создаётся |

### Ключевые поля

**`users`:**
```sql
id, login, password_hash, full_name, normalized_full_name,
organization_id, position_id, is_active,
created_at, batch_id (первый импорт)
```

**`user_course_assignments`:**
```sql
id, user_id, course_id, discipline_id, batch_id,
status (enum: assigned / in_progress / passed / failed),
assigned_at, completed_at
```

**`test_attempts`:**
```sql
id, user_id, assignment_id, attempt_number,
score, max_score, passed (bool),
started_at, finished_at
```

**`position_course_rules`:**
```sql
id, discipline_id, position_keyword, course_id, priority
```

**`import_rows`:**
```sql
id, batch_id, raw_data (JSONB), normalized_data (JSONB),
status (enum: ok / manual_review / duplicate / error),
error_message, user_id (nullable)
```

## 14. Сервисы и бизнес-логика backend

| Сервис | Назначение |
|---|---|
| `import_parser_service` | Считывает Excel, находит колонки (нечёткое совпадение заголовков), нормализует строки. |
| `import_validator_service` | Проверяет строки на критические ошибки, формирует список предупреждений. |
| `user_factory_service` | Создаёт пользователя, генерирует логин, пароль, хэш. Проверка дубликатов. |
| `login_generator_service` | Транслитерация → формирование логина → проверка уникальности → финальный логин. |
| `position_normalizer_service` | Приводит должности к нормализованному виду (lower, strip, убрать спецсимволы). |
| `assignment_engine_service` | Выбирает курсы по дисциплине и нормализованной должности через справочник правил. |
| `manual_review_service` | Собирает строки без совпадения правил, предоставляет интерфейс для ручной коррекции. |
| `test_engine_service` | Рассчитывает результат попытки, определяет pass/fail по pass_score курса, проверяет лимит попыток. |
| `export_service` | Собирает XLSX по заданным шаблонам (доступы, результаты, поток). |
| `storage_service` | Абстракция над файловым хранилищем: сохранить, получить, удалить, проверить тип/размер. |

## 15. Тесты: логика прохождения и настройки

Параметры теста задаются в таблице `tests` на уровне курса:

| Параметр | Описание | Значение по умолчанию |
|---|---|---|
| `pass_score` | Минимальный % для сдачи | 70% |
| `max_attempts` | Максимальное число попыток. `0` = без ограничений | 3 |
| `time_limit_minutes` | Ограничение по времени. `0` = без ограничения | 0 |
| `show_correct_answers` | Показывать ли правильные ответы после теста | false |

**Статусы назначения (`user_course_assignments.status`):**

- `assigned` — назначен, не начат
- `in_progress` — тест начат, попытка открыта
- `passed` — тест пройден с результатом ≥ pass_score
- `failed` — попытки исчерпаны, результат < pass_score

**Пересдача:** разрешена, если `attempt_number < max_attempts` и статус не `passed`.

## 16. Требования к UX обучающегося

- После входа пользователь сразу видит только свои назначенные курсы.
- В карточке курса: название, статус, дисциплина, кнопка «Перейти».
- Внутри курса — максимально просто: видео/embed, список материалов с кнопкой скачать, кнопка «Начать тест».
- Не нужны сложные прогресс-бары, геймификация, рейтинги или лишняя аналитика.
- Результат теста показывается сразу после сдачи: сдал / не сдал, процент, дата.
- Если есть пересдача — кнопка «Попробовать снова» (если не превышен лимит и не `passed`).

## 17. Требования к UX админки

- Админка не должна быть набором разрозненных страниц без структуры.
- Важнейшие операции: импорт → preview → корректировка → confirm → экспорт.
- Ошибки должны быть понятными: что именно не распознано и что нужно исправить.
- Таблицы должны иметь фильтры: ФИО, организация, должность, курс, статус, дата.
- На экране preview импорта строки делятся на три вкладки: ✅ OK, ⚠️ Manual review, ❌ Error.
- На вкладке Manual review должна быть возможность вручную указать курс для строки до подтверждения.

## 20. Безопасность и технические ограничения

- Пароли хранить только как hash (bcrypt, rounds ≥ 12).
- HTTPS обязателен в production (Nginx + certbot или reverse proxy).
- Приватные материалы не отдаются без авторизации — всегда через защищённый endpoint.
- Валидация загружаемых файлов: только разрешённые MIME-типы, максимальный размер (50 MB).
- Защита от дубликатов при импорте — логика в `user_factory_service` (см. раздел 11.1).
- JWT refresh токены должны инвалидироваться при смене пароля и logout.
- Все запросы к данным пользователей фильтруются по `organization_id` на уровне репозитория.
- Логи импорта хранить не удаляя — нужны для аудита.

## 23. Чего НЕ делать на MVP

- Не добавлять ЭЦП, цифровые подписи и сложную сертификацию.
- Не добавлять интерфейс для роли `client_manager` (но модель данных должна это поддерживать).
- Не делать геймификацию, рейтинги и декоративные функции.
- Не пытаться сразу решить все кейсы будущих продуктов.
- Не делать мобильное приложение — только адаптивный веб.
- Не добавлять email-уведомления (заложить абстракцию, но не реализовывать).

## 24. Краткая задача для coding-агента

Построй production-ready MVP на **FastAPI + PostgreSQL + Vue 3** для сайта и LMS-платформы по технике безопасности.

Система должна:
1. Импортировать Excel со списками сотрудников (с preview и ручной корректировкой строк без совпадений).
2. Автоматически создавать пользователей с казахской транслитерацией логина, проверкой уникальности и хэшированием пароля.
3. Назначать отдельные курсы по дисциплинам и должностям через редактируемый справочник правил.
4. Позволять обучающемуся просматривать материалы и сдавать тесты с настраиваемым pass_score и лимитом попыток.
5. Давать администратору управление курсами, правилами, прогрессом и тремя видами Excel-экспорта.
6. Хранить все данные с привязкой к организации-клиенту для будущей изоляции и роли client_manager.

Архитектура должна быть модульной, расширяемой. Бизнес-логика только в backend-сервисах. Данные разных организаций изолированы на уровне репозитория.
